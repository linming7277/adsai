本目录包含用于服务架构分析的标准化模板和工具。

## 📁 文件说明

**包含内容**:

- 依赖关系分析
- 质量评估（代码、测试、文档）
- 架构评估

- 结论

**使用方法**:
```bash
cp docs/ArchitectureReviewV1/templates/service-analysis-template.md \
   docs/ArchitectureReviewV1/<service-name>-analysis.md
```

**用途**: 完整的分析检查清单

- 后续跟踪指南

**使用方法**:
在分析过程中打开此文件，逐项检查确保分析的完整性。

**用途**: 快速参考指南

**包含内容**:
- 快速开始指南
- 评分标准
- 常见问题识别
- 常见改进建议
- 有用的命令
- Mermaid图表示例
- 质量检查清单

**使用方法**:
作为日常参考，快速查找评分标准、命令和示例。

## 🛠️ 工具

### analyze-service.sh
**位置**: `scripts/analyze-service.sh`

**功能**:
- 自动检测技术栈
- 统计代码行数
- 检查部署配置
- 分析目录结构
- 检查文档完整性
- 运行测试并获取覆盖率
- 分析依赖关系

**使用方法**:
```bash
./scripts/analyze-service.sh <service-name>
```

**示例**:
```bash
./scripts/analyze-service.sh adscenter
./scripts/analyze-service.sh proxy-pool
./scripts/analyze-service.sh browser-exec
```

**输出**:
- 技术栈信息
- 代码统计
- 文档检查结果
- 测试覆盖率
- 依赖列表
- 改进建议

## 📊 分析流程

### 标准流程（约90分钟/服务）

1. **准备** (5分钟)
   - 运行 `analyze-service.sh`
   - 复制分析模板
   - 打开检查清单

2. **服务发现** (5分钟)
   - 填写基本信息
   - 列出核心功能
   - 记录API端点

3. **代码分析** (15分钟)
   - 分析目录结构
   - 评估代码质量
   - 检查配置管理

4. **依赖分析** (10分钟)
   - 识别内部依赖
   - 列出外部依赖
   - 分析数据库使用

5. **质量评估** (15分钟)
   - 评估代码质量
   - 检查测试覆盖
   - 评估文档质量

6. **架构评估** (15分钟)
   - 识别架构模式
   - 评估设计原则
   - 识别架构问题

7. **性能和安全** (10分钟)
   - 评估性能指标
   - 检查安全措施
   - 识别潜在问题

8. **问题和建议** (15分钟)
   - 列出发现的问题
   - 制定改进建议
   - 估算工作量

9. **评分和结论** (10分钟)
   - 各维度评分
   - 计算总分
   - 撰写结论

## 🎯 评分系统

### 评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 代码质量 | 20% | 代码结构、命名、复杂度、重复度 |
| 架构设计 | 20% | 架构模式、设计原则、关注点分离 |
| 测试覆盖 | 15% | 测试完整性、测试质量 |
| 文档质量 | 10% | README、API文档、注释 |
| 安全性 | 15% | 认证授权、数据保护、输入验证 |
| 性能 | 10% | 响应时间、吞吐量、资源使用 |
| 可扩展性 | 10% | 水平扩展、状态管理、缓存策略 |

### 评分等级

- **9-10分**: 优秀 - 架构设计优良，代码质量高
- **7-8分**: 良好 - 整体质量不错，有改进空间
- **5-6分**: 中等 - 存在一些问题，需要改进
- **3-4分**: 较差 - 存在明显问题，需要重点改进
- **1-2分**: 很差 - 存在严重问题，需要重构

## 💡 最佳实践

### 分析时

1. **客观评估**: 基于事实和数据，避免主观臆断
2. **具体建议**: 提供可操作的具体建议，而非泛泛而谈
3. **优先级明确**: 按影响和紧急程度对问题排序
4. **工作量估算**: 为每个建议提供工作量估算
5. **依赖识别**: 识别改进建议之间的依赖关系

### 报告撰写

1. **使用模板**: 确保所有报告格式一致
2. **包含图表**: 使用Mermaid创建依赖图、架构图
3. **数据支撑**: 用具体数据支持评估和建议
4. **清晰简洁**: 避免冗长，突出重点
5. **审核检查**: 完成后使用检查清单审核

### 质量保证

1. **至少3个改进点**: 每个报告至少识别3个改进机会
2. **完整性检查**: 使用检查清单确保所有维度都已覆盖
3. **同行审核**: 请团队成员审核报告
4. **持续改进**: 根据反馈改进模板和流程
