## 🚨 **functions-dispatcher - 完全移除**

### **当前功能**
```yaml
功能描述: 通用HTTP调度器，通过Pub/Sub触发HTTP请求
技术实现:
  - 接收Pub/Sub消息
  - 解析HTTP调度指令 (URL, Method, Headers, Body)
  - 执行HTTP请求并返回结果
  - 支持服务间认证注入
```

### **移除理由**
1. **零业务使用**: 代码库扫描显示没有任何服务调用此dispatcher
2. **功能重复**: 直接HTTP客户端调用更简单高效
3. **架构过重**: 增加了不必要的异步调度层
4. **维护成本**: 无业务收益的技术债务

### **移除方案**
```yaml
步骤1: 停止服务部署
  - 从Cloud Run中移除functions-dispatcher服务
  - 删除相关Pub/Sub订阅

步骤2: 清理代码
  - 删除 /services/functions/ 目录
  - 移除相关配置和依赖

步骤3: 直接替代
  - 需要HTTP调用的服务直接使用标准HTTP客户端
  - 使用现有的service-client进行服务间通信
```

### **预期收益**
- **简化部署**: 减少1个微服务，降低运维复杂度
- **降低延迟**: 直接调用比异步调度更快
- **减少成本**: 减少Cloud Run服务资源使用
- **提高可靠性**: 减少故障点

## ⚠️ **projector - 简化重构**

### **当前功能**
```yaml
功能描述: 事件驱动的数据投影，实现CQRS读��型
核心事件: EventOfferCreated, EventSiterankCompleted
技术实现:
  - 接收Pub/Sub事件
  - 更新读模型数据
  - Firestore缓存（可选）
  - 事件幂等性处理
```

### **使用现状分析**
```yaml
被引用服务:
  - offer-service: 轻量使用
  - billing-service: 轻量使用
  - 其他服务: 未使用

业务价值:
  - 提供读写分离优化
  - 事件驱动架构支持
  - 查询性能提升
```

### **简化方案**
```yaml
方案A: 内联投影 (推荐)
  - 将投影逻辑移入相关业务服务
  - offer-service直接处理OfferCreated事件
  - siterank-service直接处理SiterankCompleted事件
  - 移除独立的projector服务

方案B: 保留简化 (备选)
  - 保留projector但大幅简化
  - 只处理核心业务事件
  - 移除复杂的投影逻辑
  - 专注于事件存储和基础投影
```

### **实施步骤**
```yaml
阶段1: 评估影响 (1天)
  - 分析当前投影逻辑的复杂度
  - 确认事件处理的业务依赖
  - 评估内联投影的可行性

阶段2: 重构实现 (2-3天)
  - 将投影逻辑移入业务服务
  - 更新事件处理逻辑
  - 测试事件处理正确性

阶段3: 清理移除 (1天)
  - 移除独立的projector服务
  - 清理相关配置和依赖
  - 更新文档和部署配置
```

### **预期收益**
- **简化架构**: 减少CQRS复杂度
- **降低维护**: 减少服务间依赖
- **提高性能**: 减少事件传输开销
- **便于调试**: 业务逻辑集中在一个服务
