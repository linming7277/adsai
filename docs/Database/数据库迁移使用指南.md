# 数据库迁移使用指南

**适用场景**: 项目未上线，无历史数据需要保留

## 📋 两个核心功能

### 1. 增量迁移（日常使用）
自动执行新的数据库变更

### 2. 完全重建（偶尔使用）
清空数据库并重新执行所有迁移

---

## 🚀 使用方式

### 场景1：日常开发 - 增量迁移

**触发条件**: 推送代码到main分支，且修改了迁移文件

```bash
# 1. 创建新的迁移文件
cd services/billing/migrations
# 手动创建: 000002_add_new_table.up.sql 和 .down.sql

# 2. 编写SQL
# 000002_add_new_table.up.sql
CREATE TABLE IF NOT EXISTS billing.new_table (
    id TEXT PRIMARY KEY,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

# 000002_add_new_table.down.sql
DROP TABLE IF EXISTS billing.new_table;

# 3. 提交并推送
git add services/billing/migrations/
git commit -m "feat: add new table"
git push origin main

# 4. 自动执行
# GitHub Actions 自动触发 database-migration-cloudrun.yml
# 在preview环境执行迁移
```

**工作流程**:
1. GitHub Actions检测到迁移文件变更
2. 构建db-migrator镜像
3. 创建/更新Cloud Run Job
4. 执行 `migrate up` 命令
5. 生成迁移报告

---

### 场景2：需要完全重建数据库

**使用时机**:
- 迁移文件混乱，需要从头开始
- 数据库结构有问题，需要重建
- 测试新的迁移方案

**步骤**:

#### 通过GitHub Actions（推荐）

1. 访问 GitHub Actions 页面
2. 选择工作流：`Database Migration (Cloud Run Job)`
3. 点击 `Run workflow`
4. 填写参数：
   - **environment**: 选择 `preview` 或 `production`
   - **reset_database**: 勾选 ✅（启用重置模式）
   - **confirmation_text**: 输入 `RESET DATABASE`（必须完全匹配）
5. 点击 `Run workflow` 确认执行

**自动执行流程**:
- ✅ 安全检查（验证确认文本）
- ✅ 构建迁移镜像
- ✅ 执行数据库重置（删除所有Schema）
- ✅ 顺序执行所有迁移（billing → adscenter → offer → console）
- ✅ 验证迁移结果
- ✅ 生成执行报告

**重要说明**:
- 重置模式下，迁移会**顺序执行**（max-parallel: 1）确保依赖关系
- 普通模式下，迁移会**并行执行**（max-parallel: 4）提高速度

## 📁 迁移文件规范

### 文件命名
```
services/{service}/migrations/
├── 000001_initial_schema.up.sql
├── 000001_initial_schema.down.sql
├── 000002_add_new_feature.up.sql
└── 000002_add_new_feature.down.sql
```

### 编写规范

**UP文件** (创建/修改):
```sql
-- 使用 IF NOT EXISTS 确保幂等性
CREATE SCHEMA IF NOT EXISTS billing;

CREATE TABLE IF NOT EXISTS billing.users (
    id TEXT PRIMARY KEY,
    email TEXT NOT NULL UNIQUE,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 使用 CREATE OR REPLACE 处理函数
CREATE OR REPLACE FUNCTION billing.update_timestamp()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**DOWN文件** (回滚):
```sql
-- 使用 IF EXISTS 确保安全删除
DROP FUNCTION IF EXISTS billing.update_timestamp();
DROP TABLE IF EXISTS billing.users;
DROP SCHEMA IF EXISTS billing CASCADE;
```

## ⚠️ 注意事项

### 1. 迁移文件版本号
- 必须递增（000001, 000002, 000003...）
- 不能跳过或重复
- 已执行的迁移不能修改

### 2. 幂等性
- 所有SQL必须可以重复执行
- 使用 `IF NOT EXISTS`, `IF EXISTS`, `CREATE OR REPLACE`
- 避免硬编码数据

### 3. 回滚脚本
- 每个.up.sql必须有对应的.down.sql
- 回滚脚本必须能完全撤销变更
- 测试回滚脚本是否正常工作

### 4. 数据库连接
- 所有迁移通过Cloud Run Job执行
- 使用Unix Socket连接内网数据库
- 不要直接在本地执行生产环境迁移

## 🛠️ 故障排查

### 问题1：迁移执行失败

**症状**: Job状态显示FAILED

**排查步骤**:
```bash
# 1. 查看详细日志
gcloud logging read \
  "resource.type=cloud_run_job AND resource.labels.job_name=db-migrate-*" \
  --limit=100

# 2. 检查迁移文件语法
cat services/billing/migrations/000001_initial_schema.up.sql

# 3. 检查数据库连接
gcloud sql connect autoads --user=postgres
```

**常见原因**:
- SQL语法错误
- 缺少IF NOT EXISTS
- 依赖的对象不存在
- 权限不足

### 问题2：迁移版本冲突

**症状**: `Dirty database version`

**解决方案**:
```sql
-- 连接数据库
gcloud sql connect autoads --user=postgres --database=autoads_db

-- 查看脏版本
SELECT * FROM schema_migrations WHERE dirty = true;

-- 手动修复（谨慎操作）
UPDATE schema_migrations SET dirty = false WHERE version = XXX;

-- 或者重置数据库
-- 执行reset.sh重新开始
```

### 问题3：Cloud Run Job超时

**症状**: Job执行超过10分钟被终止

**解决方案**:
```bash
# 增加超时时间
gcloud run jobs update db-migrate-billing-preview \
  --region=asia-northeast1 \
  --task-timeout=20m
```

## 🎯 快速参考

### 常用命令

```bash
# 通过GitHub CLI触发增量迁移
gh workflow run database-migration-cloudrun.yml \
  -f environment=preview

# 通过GitHub CLI触发重置并重建
gh workflow run database-migration-cloudrun.yml \
  -f environment=preview \
  -f reset_database=true \
  -f confirmation_text="RESET DATABASE"

# 查看Job日志
gcloud logging read "resource.type=cloud_run_job" --limit=50

# 连接数据库
gcloud sql connect autoads --user=postgres --database=autoads_db
```

### GitHub Actions工作流

**工作流**: `database-migration-cloudrun.yml`

| 模式 | 触发方式 | reset_database | confirmation_text | 执行方式 |
|------|----------|----------------|-------------------|----------|
| 增量迁移 | 自动（push）或手动 | false（默认） | 不需要 | 并行执行 |
| 完全重建 | 仅手动 | true | 必须输入 "RESET DATABASE" | 顺序执行 |

**参数说明**:
- `environment`: 目标环境（preview/production）
- `reset_database`: 是否重置数据库（默认false）
- `confirmation_text`: 重置确认文本（仅reset_database=true时需要）
