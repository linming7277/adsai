## 一、重构背景与目标

### 1.1 当前架构痛点

1. **前端功能不完整**
   - 缺少成熟的 SaaS 基础设施（团队管理、订阅管理、完整的 i18n）
   - UI/UX 未经系统性 SaaS 最佳实践打磨
   - 存在两套管理后台（GoFly Admin + 自研简单后台）

2. **服务冗余**
   - 部分服务功能未实现或未被使用（Workflow、Identity）
   - 维护成本高，资源浪费

3. **开发效率低**
   - 从零构建标准 SaaS 功能耗时大
   - 前端架构分散，缺少统一的设计系统

### 1.2 重构目标

#### 核心目标

✅ **功能完整性**
- 提供完整的 SaaS 基础设施（用户管理、团队协作、订阅支付）
- 统一的管理后台体验
- 完善的国际化支持（中英文）

✅ **架构清晰性**
- 前端负责"标准 SaaS 功能"，后端负责"业务逻辑与高性能计算"
- 服务精简，职责明确
- 易于维护和扩展

✅ **开发效率**
- 利用 Makerkit 提供的 80% 标准功能
- 仅需开发 20% 业务专属逻辑
- 缩短产品上市时间（Time-to-Market）

✅ **用户体验**
- 专业的 SaaS UI/UX 设计
- 流畅的认证和支付流程
- 响应式设计，支持多设备

#### 技术指标

- **服务数量**: 从 9 个精简至 7 个（-22%）
- **代码复用率**: 提升至 80%（通过 Makerkit）
- **开发周期**: 8 天完成核心迁移
- **性能目标**:
  - Landing Page LCP < 2.5s
  - Dashboard FCP < 1.5s
  - API 响应时间 P95 < 500ms

## 二、混合架构设计

### 2.1 架构原则

**混合架构 = Makerkit (Next.js) + Go 微服务**

1. **前端职责** (Makerkit)
   - 用户认证流程（注册/登录/MFA/邮箱验证）
   - 订阅管理（Stripe Checkout/Billing Portal）
   - 团队/组织管理（邀请成员、权限管理）
   - 用户设置（个人资料、密码、邮箱、组织设置）
   - Admin 后台（用户列表、组织管理、基础统计）
   - 静态内容（Landing Page、Blog、Docs、SEO）

2. **后端职责** (Go 微服务)
   - 所有核心业务逻辑（广告管理、批量访问、排名分析）
   - 高性能计算（并发任务、代理轮换、数据分析）
   - 事件驱动架构（Pub/Sub 集成）
   - 数据持久化（PostgreSQL）
   - 外部 API 集成（Google Ads API、代理服务商）
   - 业务运维工具（指标聚合、事件溯源、配置管理）

## 五、数据存储策略

### 5.1 数据分层

```yaml
Firebase Authentication:
  用途: 用户账户管理
  数据:
    - 用户凭证（Email/Password/OAuth）
    - Session 管理
    - Custom Claims（角色权限）

Firestore:
  用途: 用户配置与快速查询
  数据:
    - users/{uid}/profile（用户配置文件）
    - organizations/{orgId}（组织数据）
    - organizations/{orgId}/members/{uid}（成员关系）
    - subscriptions/{subId}（Stripe 订阅信息）
    - system_config/*（系统缓存配置）

Cloud SQL (PostgreSQL):
  用途: 业务数据持久化
  数据:
    - User 表（用户基础信息，从 Firestore 同步）
    - 业务数据（广告任务、批量访问记录、排名分析结果）
    - 计费交易记录（token_balances、token_transactions）
    - Offer 优惠数据（offers、offer_daily_kpi）
    - 事件溯源（event_store）
    - 配置管理（console_config）

Memorystore Redis:
  用途: 高速缓存与限流
  数据:
    - 限流计数器（rate_limit:*）
    - 热点数据缓存（cache:siterank:*）
    - 会话缓存（session:*）
```

### 5.2 数据同步策略

**Firestore → PostgreSQL**（用户数据同步）

```typescript
// 通过 Cloud Functions for Firebase 实现
// Trigger: firestore.document('users/{userId}').onCreate

1. 用户注册时，Makerkit 写入 Firestore users 集合
2. Cloud Function 自动触发
3. 将用户数据同步到 PostgreSQL User 表
4. 其他微服务可从 PostgreSQL 查询用户信息
```

**PostgreSQL → Firestore**（配置热更新）

```go
// Console 服务更新配置时
1. 写入 PostgreSQL console_config 表
2. 同步写入 Firestore system_config 集合（带 ETag）
3. 前端/BFF 通过 Firestore 实时获取配置变更
```

## 六、认证与权限

### 6.1 认证流程

```typescript
// 前端（Makerkit）
1. 用户通过 Firebase Auth 登录（Email/Password 或 OAuth）
2. 获取 Firebase ID Token（JWT）
3. 调用后端 API 时，在 Authorization 头携带 Token

// API Gateway
4. 验证 Firebase ID Token（通过 Firebase JWKS）
5. 注入 X-Endpoint-API-UserInfo 或 X-User-Id 头

// Go 微服务
6. 通过 pkg/auth.ExtractUserID 提取用户 ID
7. 执行业务逻辑
```

### 6.2 权限模型

#### 普通用户权限
```typescript
// Firebase Custom Claims
{
  "role": "user"
}

// 可访问路由
- /dashboard
- /adscenter/*
- /batchopen/*
- /siterank/*
- /settings/*
```

#### 管理员权限
```typescript
// Firebase Custom Claims
{
  "role": "admin"
}

// 可访问路由
- 普通用户所有路由
- /admin/*（管理后台）

// 后端权限验证
middleware.AdminOnly(handler)  // 检查 Custom Claims
```

#### 设置管理员权限

```typescript
// 通过 Firebase Admin SDK（一次性脚本）
import admin from 'firebase-admin';

async function grantAdminRole(email: string) {
  const user = await admin.auth().getUserByEmail(email);
  await admin.auth().setCustomUserClaims(user.uid, { role: 'admin' });
  console.log(`✅ Granted admin role to ${email}`);
}

// 执行
grantAdminRole('admin@autoads.dev');
```

## 九、风险与对策

### 9.1 主要风险

| 风险 | 影响 | 概率 | 对策 |
|------|------|------|------|
| Makerkit 定制化复杂 | 🟡 中 | 低 | 先使用标准功能，迭代式定制 |
| 数据同步延迟（Firestore ↔ PostgreSQL） | 🟢 低 | 低 | 使用 Cloud Functions 事件触发，延迟 < 1s |
| Stripe Webhook 幂等性 | 🟡 中 | 中 | 使用 `event.id` 去重 + 数据库唯一约束 |
| 迁移期间服务中断 | 🔴 高 | 低 | 蓝绿部署：先部署新前端到 `preview` 分支测试 |
| Console 服务负载过高 | 🟡 中 | 低 | 实施缓存策略（30 秒）+ 横向扩容 |

### 9.2 回滚策略

```yaml
如果新架构出现严重问题：
  1. 立即切换 Firebase Hosting 流量到旧前端（通过 firebase.json）
  2. API Gateway 保持不变（向后兼容）
  3. Go 微服务无需回滚（未变更）
  4. 预计回滚时间：< 5 分钟
```
