### 场景 3: 换链接 - 获取 Final URL 和 Final URL suffix

**业务场景**: 在"换链接"功能中，需要提前获取广告联盟 Offer URL 的最终落地页 URL 和查询参数（suffix），用于替换或优化广告投放的目标链接。

**测试结果**:
- ✅ **成功**: 是
- ⏱️ **耗时**: 13.0 秒
- 🔗 **Final URL**: `null/` (异常)
- 📝 **Final URL Suffix**: (空)
- 🔄 **重定向次数**: 1 次
- 🔗 **重定向链**:
  1. `https://go.dognet.com/?chid=6ab3CPGU&url=...` (原始 URL)
  2. `https://clk.tradedoubler.com/click?p=358320&a=3255845&...` (广告联盟跳转)

**分析**:
1. **成功追踪重定向**:
   - 正确识别了从 dognet.com 到 tradedoubler.com 的跳转
   - 重定向链信息完整

2. **问题 - Final URL 为 null**:
   - 返回的 `finalUrl` 是 `"null/"`，这是异常值
   - 可能原因：
     - 页面执行 `window.location.href` 时返回了 null
     - 中间页未完成最终跳转
     - JavaScript 跳转尚未执行完成

3. **Final URL Suffix 为空**:

  - 检查 `page.evaluate(() => window.location.href)` 的实现
  - 增加降级方案：使用 `page.url()` 作为备选
- 💡 对于需要多级跳转的 Offer，建议：
  - 增加稳定化时间到 15-20 秒
  - 监控重定向链长度，超过 3 次跳转可能有问题

## 具体问题排查

### 问题 1: Final URL 为什么是 `null/`？

**可能原因**:
1. **中间页未完成跳转**:
   - 当前页面是 `chromewebdata`，这是一个中间页
   - 真正的跳转可能需要用户交互（点击按钮）或更长等待时间

2. **JavaScript 执行异常**:
   ```javascript
   // 服务端代码
   const finalHref = await page.evaluate(() =>
     window.location && window.location.href || document.location.href
   )
   ```
   - 如果 `window.location.href` 为 null，拼接后变成 `"null/"`

3. **页面结构特殊**:
   - 页面可能使用了 iframe 或其他特殊结构
   - 主框架的 location 为 null

**验证方法**:
查看测试 URL 的实际行为，手动访问确认跳转流程。

### 问题 2: 为什么 60% 的场景2请求失败？

**原因**: 并发限制
- 场景2同时发起了 10 个并发请求
- 服务配置的最大并发数是 4 (`MAX_CONCURRENCY=4`)
- 超出部分立即返回 503 OVERLOADED

**解决方案**:
1. **客户端限流** (推荐):
   ```javascript
   // 使用 p-limit 控制并发
   const pLimit = require('p-limit')
   const limit = pLimit(3) // 最多3个并发

   const promises = urls.map(url =>
     limit(() => visitURL(url))
   )
   ```

2. **服务端扩容**:
   - 增加 `MAX_CONCURRENCY` 到 8-10
   - 增加 Cloud Run 实例数
   - 注意：每个并发需要一个浏览器实例，资源消耗较大

## 优化建议

### 短期优化 (1-2周)

1. **修复 Final URL 提取逻辑** 🔴 高优先级
   ```javascript
   // 添加降级方案
   let finalHref = await page.evaluate(() => window.location?.href)
   if (!finalHref || finalHref === 'null') {
     finalHref = page.url() // 降级使用 Playwright 的 page.url()
   }
   ```

2. **客户端实现并发控制** 🟡 中优先级
   - 场景2使用 p-limit 限制并发数为 3
   - 避免触发 OVERLOADED 错误

3. **增加代理IP质量监控** 🟡 中优先级
   - 记录每个代理的响应时间
   - 剔除平均响应时间 > 30秒的慢速代理
   - 优先使用高质量代理

### 中期优化 (1-2个月)

4. **增强中间页检测** 🟢 低优先级
   - 扩展中间页关键词库
   - 支持更多广告联盟的中间页识别
   - 添加页面内容语义分析

5. **实现自适应稳定化时间** 🟢 低优先级
   ```javascript
   // 根据重定向次数动态调整等待时间
   const stabilizeMs = 8000 + (redirectCount * 3000)
   ```

6. **增加失败重试机制** 🟢 低优先级
   - 目前已有 maxRetries=3
   - 增加指数退避 (exponential backoff)
   - 针对不同错误类型采用不同重试策略

### 长期优化 (3-6个月)

7. **构建代理IP池管理系统**
   - 多供应商支持
   - 代理健康度评分
   - 自动补充和淘汰
   - 按地域分配代理

8. **实现智能路由**
   - 根据 Offer URL 的域名选择最优代理
   - 缓存成功的域名-代理映射
   - 减少试错次数

9. **增加监控和告警**
   - 成功率监控
   - 响应时间监控
   - 代理质量监控
   - Offer 可用性趋势分析
