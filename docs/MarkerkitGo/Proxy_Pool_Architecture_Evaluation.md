# 代理池架构评估报告

**评估日期**: 2025-10-02
**当前问题**: Worker 分布式代理获取导致 API 限流

---

## 一、当前架构分析

### 1.1 现状

**架构**: 每个 Worker 实例独立管理代理池

```
┌─────────────────────────────────────────────────┐
│  Proxy Provider API                              │
│  (iprocket.io)                                   │
└────────────┬────────────────────────────────────┘
             │
             │  每个 Worker 独立请求
             │
    ┌────────┼────────┬────────┬────────┐
    │        │        │        │        │
    ▼        ▼        ▼        ▼        ▼
┌────────┐ ┌────────┐ ┌────────┐ ... ┌────────┐
│Worker 1│ │Worker 2│ │Worker 3│     │Worker N│
│SmartPxy│ │SmartPxy│ │SmartPxy│     │SmartPxy│
│Pool    │ │Pool    │ │Pool    │     │Pool    │
└────────┘ └────────┘ └────────┘     └────────┘
  5-20 个实例并发运行
```

**配置**:
- Worker 实例数: 5-20（自动扩展）
- 每个实例的代理池: SmartProxyPool
- 每次请求代理数: 50 个（MIN_POOL_SIZE）
- 缓存 TTL: 5 分钟
- 全局锁窗口: 5 秒

### 1.2 发现的问题

#### 🔴 问题 1: API 并发请求风暴
**现象**: 
- 测试启动后 5 秒内触发限流
- 87.2% 的代理获取请求失败
- 只成功获取 12 次（共 600 个代理）

**原因**:
1. **冷启动风暴**: 5 个 Worker 实例同时启动，缓存都是空的
2. **并发请求**: 5 实例 × 10 并发 = 50 个任务同时请求代理
3. **缓存未命中**: 每个任务都触发 `fetchProxies(50)`
4. **API 限流**: 短时间内 100+ 请求超过提供商限制

**时间线**:
```
14:09:00 - 400 个任务发布到队列
14:09:05 - 5 个 Worker 实例启动
14:24:46 - 开始获取代理（成功）
14:24:51 - 5 秒后开始出现失败
14:25:24 - 最后一次成功
14:25:25+ - 持续失败（限流生效）
```

#### 🔴 问题 2: 代理池隔离导致浪费
**现象**: 
- 每个 Worker 维护独立的 50 个代理池
- 5-20 实例 = 250-1000 个代理被分散持有
- 实际使用率低（单个任务只用 1 个代理）

**资源浪费**:
```
理论需求: 400 任务 × 1 代理 = 400 个代理
实际获取尝试: 100+ 请求 × 50 = 5000+ 个代理请求
成功获取: 12 × 50 = 600 个代理
利用率: < 10%（大部分代理空闲或超时）
```

#### 🔴 问题 3: 无法协调和限流
**现象**:
- Worker 之间无法感知彼此的请求
- 无法实现全局限流控制
- 无法在 Worker 间共享代理

**后果**:
- 重复获取相同的代理
- 无法应对提供商的全局限流
- 扩展性受限（越多 Worker = 越多并发冲突）

## 二、架构方案对比

### 方案 A: 现状（分布式代理池）

**架构**: 每个 Worker 独立管理代理池

**优点**:
- ✅ 简单，无额外依赖
- ✅ Worker 独立运行，不依赖外部服务
- ✅ 故障隔离（一个 Worker 故障不影响其他）

**缺点**:
- ❌ API 并发请求风暴（已证实）
- ❌ 代理资源浪费严重
- ❌ 无法全局限流和协调
- ❌ 扩展性差（Worker 越多，冲突越严重）
- ❌ 无法共享代理（缓存隔离）

**适用场景**: 
- Worker 实例少（1-3 个）
- 代理 API 无限流限制
- 任务量小（< 100/小时）

---

### 方案 B: 统一代理池服务（推荐）

**架构**: 独立的代理池管理服务

```
┌─────────────────────────────────────────────────┐
│  Proxy Provider API                              │
│  (iprocket.io)                                   │
└────────────┬────────────────────────────────────┘
             │
             │  统一请求 + 限流
             ▼
┌────────────────────────────────────────────────┐
│  Proxy Pool Manager Service                    │
│  (独立 Cloud Run 服务)                          │
│                                                 │
│  - 统一获取代理（批量 200 个）                  │
│  - 全局限流（max 1 req/10s）                   │
│  - Redis 缓存共享（TTL 15 min）                │
│  - 健康检查和淘汰                               │
│  - API: GET /proxy?url=xxx                     │
└────────────┬────────────────────────────────────┘
             │
             │  Worker 请求代理
             │
    ┌────────┼────────┬────────┬────────┐
    │        │        │        │        │
    ▼        ▼        ▼        ▼        ▼
┌────────┐ ┌────────┐ ┌────────┐ ... ┌────────┐
│Worker 1│ │Worker 2│ │Worker 3│     │Worker N│
│        │ │        │ │        │     │        │
└────────┘ └────────┘ └────────┘     └────────┘
```

**核心特性**:
1. **统一入口**: 所有代理请求通过 Proxy Manager
2. **全局限流**: 控制对提供商 API 的请求频率
3. **Redis 共享**: Worker 间共享代理池
4. **批量获取**: 一次获取 200 个代理，长期缓存
5. **健康管理**: 统一追踪代理健康度
6. **智能分配**: 根据 URL 和 Worker 负载智能分配

**技术实现**:
```javascript
// Proxy Pool Manager API
// GET /api/proxy?targetUrl=xxx

class ProxyPoolManager {
  constructor() {
    this.redis = createRedisClient()
    this.rateLimiter = new RateLimiter(1, 10000) // 1 req/10s
  }

  async getProxy(targetUrl) {
    // 1. 从 Redis 获取可用代理
    let proxy = await this.redis.lPop('proxy:available')
    
    if (!proxy) {
      // 2. 代理池为空，触发补充（限流保护）
      await this.rateLimiter.acquire()
      await this.refillProxies(200)
      proxy = await this.redis.lPop('proxy:available')
    }
    
    // 3. 标记为使用中
    await this.redis.setex(`proxy:${proxy}:locked`, 60, targetUrl)
    
    return proxy
  }

  async refillProxies(count) {
    const proxies = await fetchFromProvider(count)
    await this.redis.rPush('proxy:available', ...proxies)
    console.log(`Refilled ${proxies.length} proxies`)
  }

  async releaseProxy(proxy, success) {
    await this.redis.del(`proxy:${proxy}:locked`)
    
    if (success) {
      // 代理健康，放回池中
      await this.redis.rPush('proxy:available', proxy)
    } else {
      // 代理失败，淘汰
      console.log(`Proxy ${proxy} marked as failed`)
    }
  }
}
```

**优点**:
- ✅ 解决 API 并发风暴（全局限流）
- ✅ 代理高效利用（共享池）
- ✅ 扩展性强（Worker 数量不影响代理获取）
- ✅ 统一健康管理
- ✅ 减少 API 调用（批量获取）
- ✅ 可监控、可管理

**缺点**:
- ⚠️ 增加依赖（Redis + 独立服务）
- ⚠️ 单点故障风险（需要高可用部署）
- ⚠️ 网络延迟（Worker ↔ Manager）

**成本**:
- Cloud Run: 1 实例，256MB，~$5/月
- Redis (Memorystore): 基础版 1GB，~$35/月
- 总计: ~$40/月

### 方案 C: 混合方案（过渡方案）

**架构**: Worker 本地缓存 + 协调机制

```
┌─────────────────────────────────────────────────┐
│  Proxy Provider API                              │
└────────────┬────────────────────────────────────┘
             │
    ┌────────┼────────┐
    │  全局限流器      │
    │  (Firestore)    │  1 req/10s
    └────────┬────────┘
             │
    ┌────────┼────────┬────────┬────────┐
    │        │        │        │        │
    ▼        ▼        ▼        ▼        ▼
┌────────┐ ┌────────┐ ┌────────┐ ... ┌────────┐
│Worker 1│ │Worker 2│ │Worker 3│     │Worker N│
│SmartPxy│ │SmartPxy│ │SmartPxy│     │SmartPxy│
│+ Lock  │ │+ Lock  │ │+ Lock  │     │+ Lock  │
└────────┘ └────────┘ └────────┘     └────────┘
```

**核心改进**:
1. **分布式锁**: Firestore 实现全局限流
2. **延长缓存**: TTL 从 5 分钟 → 15 分钟
3. **减少批量**: MIN_POOL_SIZE 从 50 → 20
4. **重试机制**: 失败后指数退避重试
5. **预热协调**: Worker 启动时错峰获取

**实现**:
```javascript
class GlobalRateLimiter {
  constructor() {
    this.firestore = getFirestore()
  }

  async acquire() {
    const lockDoc = this.firestore.doc('locks/proxy-fetch')
    
    // 尝试获取锁（10 秒窗口）
    const now = Date.now()
    const result = await lockDoc.set({
      timestamp: now,
      holder: process.env.K_REVISION
    }, { merge: true })
    
    const data = await lockDoc.get()
    
    // 检查是否在窗口内
    if (now - data.timestamp < 10000) {
      // 有其他 Worker 在获取代理，等待
      await sleep(Math.random() * 5000 + 5000) // 5-10s
      return this.acquire() // 递归重试
    }
    
    return true
  }
}
```

**优点**:
- ✅ 快速实施（最小改动）
- ✅ 无需新服务
- ✅ 缓解并发问题
- ✅ 成本低

**缺点**:
- ⚠️ 仍有冲突可能（Firestore 非强一致性）
- ⚠️ 无法共享代理（缓存仍隔离）
- ⚠️ 性能有损（分布式锁延迟）
- ⚠️ 治标不治本
