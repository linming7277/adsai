## 🚫 严格禁止操作清单

### 预发环境禁止操作
```bash
🚫 绝对禁止在预发环境执行：
- CREATE TABLE / ALTER TABLE / DROP TABLE
- CREATE INDEX / DROP INDEX
- INSERT / UPDATE / DELETE / TRUNCATE
- 任何修改表结构的DDL操作
- 任何修改数据的DML操作

✅ 预发环境只允许：
- SELECT 查询（只读权限）
- Schema查询（\dt, \di）
- 连接状态检查
- 性能监控查询
```

### 生产环境限制操作
```bash
⚠️ 需要严格审批的操作：
- 所有DDL操作
- 大批量数据修改（>100行）
- 索引创建和删除
- 表结构变更
- 用户权限变更

✅ 相对安全的操作：
- 使用IF NOT EXISTS语法的DDL
- 单条记录的DML操作
- 只读查询操作
- 监控和告警查询
```

## 📚 知识培训模块

### 模块1: 环境风险理解 (1小时)

#### 1.1 共享数据库环境风险
```markdown
**学习目标**: 理解为什么共享环境风险极高

**关键知识点**:
1. 预发和生产环境数据完全一致
2. 预发测试操作直接影响生产服务
3. 无法独立验证迁移效果
4. 回滚影响范围更大

**实际案例**:
- 某团队在预发环境测试删除操作，意外删除了生产数据
- 预发环境的性能测试导致生产服务响应变慢
- 预发环境的配置变更影响了生产服务稳定性
```

#### 1.2 风险缓解策略
```markdown
**核心策略**:
1. 生产环境直接执行（但极其谨慎）
2. 完整的备份和回滚机制
3. 分阶段执行和验证
4. 实时监控和告警

**关键措施**:
- 操作前完整备份
- 使用IF NOT EXISTS语法
- 实时性能监控
- 15分钟快速回滚目标
```

### 模块3: 监控和告警 (1.5小时)

#### 3.1 关键监控指标
```bash
# 实时监控脚本使用
# 1. 启动性能监控
/tmp/performance_monitor.sh &

# 2. 启动错误监控
/tmp/error_monitor.sh &

# 3. 检查监控结果
tail -f /tmp/db_performance_*.log
tail -f /tmp/db_error_monitor_*.log

# 4. 手动查询监控
-- 连接数监控
SELECT state, count(*) FROM pg_stat_activity GROUP BY state;

-- 性能监控
SELECT mean_time, calls FROM pg_stat_statements ORDER BY mean_time DESC LIMIT 5;

-- 锁等待监控
SELECT blocked_locks, locktype FROM pg_locks WHERE NOT granted;
```

#### 3.2 告警阈值配置
```yaml
# 关键告警阈值
connections:
  warning: 50  # 活跃连接数
  critical: 80  # 活跃连接数

performance:
  slow_queries: 10    # 慢查询数
  avg_time: 500      # 平均响应时间(ms)
  error_rate: 0.5    # 错误率(%)

database:
  size_growth: 10      # 数据库大小增长(GB)
  connection_errors: 5 # 连接错误数
```

### 模块4: 应急响应 (1.5小时)

#### 4.1 应急响应流程
```markdown
**响应时间目标**:
- 问题检测: < 2分钟
- 决策时间: < 3分钟
- 执行时间: < 10分钟
- 总计时间: < 15分钟

**决策树**:
1. 检测到问题 → 2分钟内识别根因
2. 评估影响范围 → 3分钟内确定处理策略
3. 执行恢复操作 → 10分钟内完成恢复
4. 验证恢复结果 → 确认服务正常
```

## 📋 操作标准和流程

### 标准操作流程 (SOP)

#### 操作前检查清单
```markdown
□ 环境确认
  □ 确认当前环境（预发/生产）
  □ 检查数据库连接状态
  □ 确认备份策略有效
  □ 验证监控工具正常

□ 业务时间窗口
  □ 确认业务低峰期
  □ 协调相关团队
  □ 设置预期完成时间
  □ 准备应急联系人

□ 技术准备
  □ 检查脚本版本
  □ 验证参数配置
  □ 测试数据库连接
  □ 确认备份路径
```

#### 执行过程标准
```markdown
1. **准备阶段** (5分钟)
   - 登录目标环境
   - 检查当前状态
   - 启动监控工具
   - 确认备份完成

2. **执行阶段** (10分钟)
   - 按步骤执行操作
   - 实时监控状态
   - 记录操作日志
   - 检查执行结果

3. **验证阶段** (5分钟)
   - 验证功能正常
   - 检查性能指标
   - 确认服务状态
   - 生成执行报告
```

#### 操作后检查清单
```markdown
□ 功能验证
  □ 所有功能测试通过
  □ 数据完整性确认
  □ 服务集成正常
  □ 性能指标达标

□ 监控确认
  □ 监控工具运行正常
  □ 告警规则配置正确
  □ 日志记录完整
  □ 错误通知正常

□ 文档更新
  □ 操作记录完整
  □ 执行结果已记录
  □ 问题已记录和跟踪
  □ 经验教训已总结
```

### 问题处理流程

#### 标准问题处理
```markdown
1. **问题识别**
   - 监控系统自动告警
   - 用户反馈系统异常
   - 主动检查发现问题

2. **问题分析**
   - 收集错误日志和监控数据
   - 分析根本原因
   - 评估影响范围

3. **问题处理**
   - 执行标准处理流程
   - 实时监控处理过程
   - 记录处理结果

4. **问题验证**
   - 验证问题已解决
   - 监控系统恢复正常
   - 通知相关人员
   - 更新问题记录
```

#### 紧急事件处理
```markdown
1. **立即响应** (< 2分钟)
   - 启动应急响应团队
   - 评估影响范围
   - 通知相关方
   - 准备回滚方案

2. **快速处理** (< 10分钟)
   - 执行应急预案
   - 监控处理过程
   - 实时通报进展
   - 必要时执行回滚

3. **恢复验证** (< 5分钟)
   - 验证系统恢复
   - 检查数据完整性
   - 确认服务正常
   - 通知恢复完成

4. **事后分析**
   - 生成事故报告
   - 分析根本原因
   - 制定改进措施
   - 更新操作流程
```
