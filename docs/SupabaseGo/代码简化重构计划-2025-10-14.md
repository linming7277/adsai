## 二、Console服务非核心功能评估

### 可移除的非核心功能 ❌

#### 1. **Telemetry Forwarder** (268行)
- **文件**: `internal/handlers/telemetry_forwarder.go`
- **功能**: 转发Web Vitals到New Relic/Datadog/Logflare
- **评估**: ❌ 非核心功能,属于监控增强
- **影响**: 仅用于性能监控,无业务依赖
- **建议**: 删除,改用GCP Cloud Monitoring

#### 2. **Health Aggregate** (211行)
- **文件**: `internal/handlers/health_handlers.go`
- **功能**: 聚合所有微服务健康检查
- **评估**: ⚠️ 可选功能,营销/监控性质
- **影响**: `/api/health`端点被前端或外部监控使用
- **建议**: 简化为仅检查本服务,或移至独立监控服务

#### 3. **Token Analytics** (215行)
- **文件**: `internal/handlers/token_analytics.go`
- **功能**: Token消费趋势分析、Top消费者排行
- **评估**: ⚠️ 增强功能,非核心业务
- **影响**: 管理后台数据分析面板
- **建议**: 移至独立Analytics服务或简化为基础统计

#### 4. **Marketing Cache** (散落在多个文件)
- **功能**: 缓存营销数据汇总
- **评估**: ❌ 非核心功能
- **建议**: 删除营销相关缓存逻辑

- ✅ 用户管理 (users_handlers.go)
- ✅ Token管理 (tokens_handlers.go)
- ✅ 任务管理 (tasks.go)

## 四、重构原则和约束

### 强制约束
- ✅ 单文件<300行(重构后略超可接受,不超过320行)
- ✅ Frontend: page.tsx<100行(仅组装), hooks<150行, components<200行
- ✅ Backend: handler<200行, service<300行, repository<200行
- ✅ main.go必须<200行(仅启动逻辑)

### 拆分策略
1. **按领域拆分**: 不同业务领域独立文件
2. **按层次拆分**: handler → service → repository
3. **按功能拆分**: CRUD操作可独立为子处理器
4. **提取共享**: 通用逻辑提取到pkg/目录

### 命名规范
```
// 好的拆分示例
users_handlers.go (379行) → 可拆分为:
├── users_list.go      (获取用户列表)
├── users_detail.go    (用户详情)
├── users_actions.go   (用户操作)
└── users_stats.go     (用户统计)
```

## 六、风险评估

### 高风险点 ⚠️

1. **Console服务功能移除**
   - 风险: 前端或外部监控可能依赖被删除的端点
   - 缓解: 先检查Frontend调用,确认无依赖后再删除

2. **大文件拆分**
   - 风险: 引入import循环依赖
   - 缓解: 遵循分层架构,避免相互引用

3. **测试覆盖**
   - 风险: 重构后遗漏测试用例
   - 缓解: 重构前先运行测试,确保绿色状态

### 回滚计划

```bash
# 如果重构出现问题,回滚步骤:
1. git revert <commit-hash>
2. 推送回滚commit触发重新部署
3. 在本地修复问题后重新提交
```
