# AutoAds 无组织模式重构方案

> **方案版本**: v1.0
> **创建日期**: 2025-10-10
> **负责人**: AutoAds Core Team
> **目标**: 彻底移除组织概念，简化为 User-centric 架构

---

## 📋 执行摘要

### 当前问题
1. 前端路由保留 `[organization]` 动态参数，但数据库无对应表
2. 代码中存在大量"假装有组织"的逻辑（RPC调用、Context、查询）
3. URL 冗长且无意义（如 `/dashboard/uuid/offers`）
4. 维护成本高，性能浪费

### 目标状态
- ✅ 路由简化：`/dashboard/offers`, `/dashboard/tasks`
- ✅ 代码清晰：移除所有组织相关抽象
- ✅ 性能优化：减少数据库查询和 Context 层级
- ✅ 用户体验：URL 简洁易懂

### 工作量评估
- **总工时**: 2-3天
- **风险等级**: 中（需充分测试）
- **回滚难度**: 低（通过重定向保证兼容）

## 🎯 方案论证

### 1. 数据库事实验证

```sql
-- 核心业务表结构
public.offers (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES auth.users(id),
  ...
)

public.tasks (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES auth.users(id),
  ...
)

public.ads_connections (
  user_id UUID NOT NULL,
  ...
)
```

**结论**: 所有表基于 `user_id`，无 `organization_id` 字段

### 2. RLS 策略验证

```sql
-- Offers RLS
POLICY "offers_select_own" FOR SELECT
  USING (auth.uid() = user_id)

-- Tasks RLS
POLICY "tasks_select_own" FOR SELECT
  USING (auth.uid() = user_id)
```

**结论**: 权限控制基于 `user_id`，组织概念多余

### 3. 业务模型验证

- **当前用户数**: 2个（测试账号）
- **组织表**: 不存在
- **成员表**: 不存在
- **邀请功能**: 存在代码但无法工作

**结论**: 系统本质是个人 SaaS，非团队协作工具

## 📐 架构设计

### 路由结构对比

#### 当前架构（多组织遗留）
```
/dashboard                          -> 重定向逻辑
/dashboard/[organization]/offers    -> Offers 列表
/dashboard/[organization]/tasks     -> Tasks 列表
/dashboard/[organization]/ads-center -> 广告中心
/dashboard/[organization]/settings/profile -> 个人设置
/dashboard/[organization]/settings/organization -> 组织设置（无法工作）
```

#### 目标架构（无组织）
```
/dashboard                -> Dashboard 首页
/dashboard/offers         -> Offers 列表
/dashboard/tasks          -> Tasks 列表
/dashboard/ads-center     -> 广告中心
/settings/profile         -> 个人设置
/settings/tokens          -> Token 管理
/settings/subscription    -> 订阅管理
```

### 代码结构对比

#### 当前结构
```
app/
├── dashboard/
│   ├── [organization]/          # 动态路由
│   │   ├── offers/
│   │   ├── tasks/
│   │   ├── ads-center/

├── organizations/                # 组织相关逻辑

```

#### 目标结构
```
app/
├── dashboard/
│   ├── page.tsx                  # Dashboard 首页
│   ├── offers/                   # 直接路由
│   ├── tasks/

│   ├── profile/
│   ├── tokens/
│   └── subscription/

└── user/                         # 用户相关逻辑（移除 organizations/）
```

### Phase 4: 兼容性中间件（0.5天）

#### 目标
添加重定向逻辑，确保旧链接继续工作

#### 任务清单
- [ ] T4.1: 创建或更新 `middleware.ts`
  ```typescript
  export function middleware(request: NextRequest) {
    const { pathname } = request.nextUrl;

    // 重定向旧组织路由到新路由
    const orgRouteMatch = pathname.match(/^\/dashboard\/([a-f0-9-]{36})\/(.+)$/);
    if (orgRouteMatch) {
      const [, , path] = orgRouteMatch;
      return NextResponse.redirect(
        new URL(`/dashboard/${path}`, request.url)
      );
    }

    // 重定向旧 settings 路由
    const settingsMatch = pathname.match(/^\/dashboard\/([a-f0-9-]{36})\/settings\/(.+)$/);
    if (settingsMatch) {
      const [, , path] = settingsMatch;
      return NextResponse.redirect(
        new URL(`/settings/${path}`, request.url)
      );
    }

    return NextResponse.next();
  }
  ```

- [ ] T4.2: 配置 middleware matcher

## 🔄 回滚方案

### 回滚触发条件
1. 核心功能无法正常工作（Offers/Tasks 页面报错）
2. 数据查询失败率 > 5%
3. 用户反馈重大体验问题

### 回滚步骤

#### 快速回滚（5分钟）
```bash
# 1. 回滚到上一个稳定版本
git revert HEAD~3..HEAD  # 假设最近3个commit是重构内容

# 2. 重新部署
git push origin main

# 3. 等待 GitHub Actions 自动部署
gh run watch
```

#### 完整回滚（30分钟）
1. 恢复 `[organization]` 路由目录
2. 恢复 `lib/organizations/` 代码
3. 移除 middleware 重定向逻辑
4. 恢复导航链接
5. 运行完整测试
6. 部署到 Preview 环境验证
7. 部署到生产环境

### 回滚风险
- ✅ **低**: 数据库无变更，无数据丢失风险
- ✅ **低**: 通过重定向保证旧链接兼容
- ⚠️ **中**: 用户可能已收藏新链接（通过反向重定向解决）
