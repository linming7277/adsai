## 📋 执行摘要

### 核心变更

**架构简化**:
```
❌ 旧架构: /dashboard/[organization]/offers  (32个字符)
✅ 新架构: /dashboard/offers                 (17个字符, -47%)
```

**关键决策**:
1. ✅ **保留 `/dashboard` 前缀** - 符合应用架构设计 (appPrefix)
2. ✅ **直接切换，无需重定向** - 项目未上线，无历史包袱
3. ✅ **彻底移除组织代码** - 减少 3000+ 行冗余代码

### 为什么保留 `/dashboard` 前缀？

#### 架构设计原因

**配置文件证据** (`configuration.ts:65-66`):
```typescript
paths: {
  appPrefix: '/dashboard',  // 应用区域统一前缀
  appHome: '/dashboard',    // 登录后默认首页
}
```

**路由分层结构**:
```
/                           # 公开营销页面
├── (site)/                 # 营销内容区域
│   ├── page.tsx           # Landing Page
│   ├── pricing/           # 定价页
│   ├── about/             # 关于我们
│   └── blog/              # 博客
├── auth/                   # 认证区域（未登录可访问）
│   ├── sign-in/
│   └── sign-up/
└── dashboard/              # 应用区域（需登录）✅
    ├── page.tsx           # Dashboard 首页
    ├── offers/            # 业务功能
    ├── tasks/
    └── ads-center/
```

**设计理念**:
1. **职责分离**: 公开内容 vs 应用功能 vs 认证流程
2. **权限控制**: `/dashboard/*` 统一应用认证中间件
3. **URL 语义**: 用户明确知道当前在"应用内部"
4. **行业惯例**: 大部分 SaaS 产品采用此结构（Vercel, Notion, Linear 等）

#### 如果改为 `/offers` 会怎样？

**潜在问题**:
```
/offers              # ❌ 与营销页面混淆（是产品介绍还是用户数据？）
/blog/offers-guide   # ❌ URL 冲突风险
/(site)/offers       # ❌ Next.js 路由组冲突
```

**维护问题**:
- ❌ 认证中间件需单独配置每个路由
- ❌ 导航组件需判断当前是否在应用区
- ❌ 未来扩展困难（如添加 `/reports`, `/analytics` 等）

**结论**: **保留 `/dashboard` 前缀符合架构最佳实践**


```typescript
'use client';

import { createContext, useContext } from 'react';
import type { User } from '@supabase/supabase-js';

interface UserContextValue {
  user: User;
  isLoading: boolean;
}

const UserContext = createContext<UserContextValue | undefined>(undefined);

export function UserProvider({
  children,
  user
}: {
  children: React.ReactNode;
  user: User;
}) {
  return (
    <UserContext.Provider value={{ user, isLoading: false }}>
      {children}
    </UserContext.Provider>
  );
}

export function useUser() {
  const context = useContext(UserContext);
  if (!context) {
    throw new Error('useUser must be used within UserProvider');
  }
  return context;
}
```

- 保留详细的错误日志

**应急方案**:
- 临时回退到 Client-Side 数据获取
- 使用 `useEffect` + `useState` 加载数据

---

#### 2. 遗漏的组织依赖导致运行时错误（概率: 中，影响: 中）

**症状**: 某个页面或功能意外调用 `useCurrentOrganization()`

**缓解措施**:
- 使用 IDE 全局搜索 `useCurrentOrganization`
- 使用 ESLint 规则禁止导入已删除的模块
- 完整功能测试覆盖所有页面

**应急方案**:
- 创建 Mock `useCurrentOrganization` 返回 `null`
- 逐步修复遗漏点
