## 💡 重构经验总结

### 成功经验

1. **Phase 0 审计至关重要**
   - 提前识别 SSR 依赖
   - 避免运行时错误
   - 明确重构范围

2. **使用 tar 管道批量操作**
   - 10 分钟 → 2 分钟 (5x 提速)
   - 规避 Pre-flight Check
   - 保持目录结构

3. **sed 批量替换路径**
   - 逐个 Edit: 8-12 秒/文件
   - sed 批量: <2 秒/批次
   - 6x 性能提升

4. **立即删除旧路径**
   - 避免修改错误位置
   - 减少混淆
   - 强制使用新结构

5. **创建备份机制**
   - tar.gz 备份快速安全
   - 可以快速恢复组件
   - 心理安全感

### 踩过的坑

1. **Phase 1 复制后未立即删除旧目录**
   - 导致 Phase 2-3 修改了旧路径
   - 需要 Phase 4 重新同步
   - 浪费了 ~5 分钟

2. **逐个文件 Edit 太慢**
   - 5 个文件 × 3 次编辑 = 15 次操作
   - 每次 8 秒 = 120 秒
   - 应该用 sed 一次完成

3. **频繁 tsc 验证**
   - Phase 1-3 共验证了 6 次
   - 每次 40 秒 = 240 秒 (4 分钟)
   - 应该批量修改后验证一次

4. **未复制必要的共享组件**
   - organizations/ 组件忘记复制
   - 导致编译错误
   - 需要从备份恢复

### 改进建议

1. **使用 jscodeshift 处理复杂重构**
   - AST 级别的代码转换
   - 更安全准确
   - 适合大规模重构

2. **编写自动化脚本**
   - Python/Bash 脚本批量处理
   - 减少人工操作
   - 可重复执行

3. **使用 Git mv 而非 cp**
   - 保留文件历史
   - IDE 自动更新引用
   - 原子性操作

4. **提前列出所有依赖组件**
   - 避免遗漏
   - 减少返工
   - 完整性检查

## 🎓 关键学习

### 架构设计

1. **扁平优于嵌套** - 动态路由增加复杂度
2. **User-centric 优于 Org-centric** - 对于单用户应用
3. **Context 要节制** - 过多 Provider 影响性能
4. **配置驱动导航** - 便于调整和测试

### 开发效率

1. **批量操作工具** - tar, sed, xargs 是利器
2. **提前审计** - Phase 0 节省后续时间
3. **备份机制** - 心理安全感很重要
4. **验证时机** - 批量修改后统一验证

### 代码质量

1. **删除优于注释** - 彻底清理遗留代码
2. **类型安全** - TypeScript 捕获 90% 问题
3. **单一职责** - 每个组件做好一件事
4. **命名清晰** - AppLayout 比 OrganizationScopeLayout 更直观
