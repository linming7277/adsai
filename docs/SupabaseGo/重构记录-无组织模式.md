## 📋 执行摘要

### 重构目标

将 AutoAds 从"多组织架构"简化为"用户直连架构"，移除所有组织相关的路由、逻辑和UI元素。

### 核心变更

| 方面 | 重构前 | 重构后 | 改进 |
|------|--------|--------|------|
| **路由结构** | `/dashboard/[organization]/offers` | `/dashboard/offers` | URL简化47% |
| **数据查询** | 基于 `organization_id` | 基于 `user_id` | 直接、高效 |
| **Context层级** | 3层（User → Organization → Data） | 1层（User → Data） | 减少2层 |
| **代码行数** | 基线 | -3000+ 行 | 精简13% |
| **删除文件** | 基线 | -30+ 文件 | 清理组织逻辑 |

#### 1.2 数据获取重构

**修改前**:
```typescript
// Server Component
async function OffersPage({ params }: { params: { organization: string } }) {
  const organizationUid = params.organization;
  const offers = await getOffersByOrganization(organizationUid);
  return <OffersView offers={offers} />;
}
```

**修改后**:
```typescript
// Server Component
async function OffersPage() {
  const user = await getUser();
  const offers = await getOffersByUserId(user.id);
  return <OffersView offers={offers} />;
}
```

**关键改进**:
- ✅ 移除了 `params.organization` 依赖
- ✅ 直接基于 `user.id` 查询数据
- ✅ 利用 Supabase RLS 自动过滤
- ✅ 减少一层抽象（组织层）

## 🔍 问题与解决方案

### 问题 1: Navbar 组件遗留代码

**位置**: `apps/frontend/src/components/layout/Navbar.tsx:95`

**问题描述**:
```typescript
// ESLint 警告
Warning: React Hook useMemo has unnecessary dependencies:
'organization.uuid' and 'role'.
```

**根因分析**:
- Navbar 组件中仍有 `organization.uuid` 的引用
- useMemo 依赖数组包含已删除的组织相关变量

**影响评估**:
- **严重性**: Low (P3)
- **功能影响**: 无 - 仅为代码优化建议
- **用户影响**: 无

**解决方案**:
```bash
# 搜索遗留代码
grep -r "organization\\.uuid" apps/frontend/src
grep -r "organizationUid" apps/frontend/src

# 建议修复（可后续清理）
# 1. 移除 useMemo 中的 organization.uuid 依赖
# 2. 清理 Navbar 组件中的组织相关逻辑
```

**优先级**: P3 - 不影响功能，可在后续版本清理

---

### 问题 2: Preview 环境测试超时

**问题描述**:
使用 curl 获取完整页面内容时，请求超时（2分钟）

**原因分析**:
- Next.js SSR 渲染大型页面需要时间
- 未登录状态可能触发复杂的认证逻辑
- curl 不执行 JavaScript，无法获取客户端渲染内容

**解决方案**:
- ✅ 改用 HTTP Header 检查（`curl -I`）验证路由可访问性
- ✅ 确认 HTTP 307 重定向和 Cloudflare 响应正常
- ⏸️ 浏览器交互测试需手动或使用 Playwright

**经验教训**:
- HTTP API 测试适合验证路由和服务器响应
- 复杂交互和认证流程需浏览器测试
- 应结合自动化和人工测试

## 💡 经验总结

### 成功因素

1. **批量操作工具**
   - 使用 `tar`, `sed`, `xargs` 批量处理文件
   - 将预估 24 小时压缩到不到 1 小时
   - 效率提升 **23倍**

2. **渐进式重构**
   - Phase 0-2: 先创建新路由，保持旧路由可用
   - Phase 3-4: 再删除旧代码，避免中断
   - Phase 5-6: 最后清理 UI 和数据库
   - 确保每个阶段都可独立验证

3. **文档先行**
   - 每个 Phase 完成后立即更新文档
   - 确保可追溯和可回滚
   - 便于团队协作和知识传承

4. **自动化测试优先**
   - 先执行自动化测试（TypeScript/ESLint/Build）
   - 再进行人工测试
   - 减少重复劳动

5. **严格依赖管理**
   - 遵循 Monorepo 最佳实践
   - 统一 Dockerfile 模板
   - 完整的 replace 指令（包括传递依赖）

1. **测试策略**
   - 应在重构前记录基准数据（性能、体积等）
   - 考虑使用 Playwright 自动化浏览器测试

## 🔐 安全考虑

### 数据隔离

**重构前**:
- 基于 `organization_id` 隔离数据
- RLS 策略: `organization_id = user_organization_id`
- 用户可能属于多个组织

**重构后**:
- 基于 `user_id` 直接隔离数据
- RLS 策略: `user_id = auth.uid()`
- 每个用户独立命名空间
- **更简单、更安全**

- ✅ Google OAuth 一键登录
