### 2. 为什么需要Google Ads OAuth Scopes？

**问题**: 为什么需要 `adwords` 的 OAuth Scopes？

**答案**: Google Ads OAuth用于**授权访问用户的Google Ads账号数据**

#### Google Ads API的工作原理

```
┌─────────────┐                 ┌──────────────┐                 ┌─────────────┐
│  AdsAI    │ OAuth授权请求   │   Google     │  授权码         │   AdsAI   │
│  前端页面   │────────────────>│ OAuth Server │────────────────>│  后端服务   │
└─────────────┘                 └──────────────┘                 └─────────────┘
      │                                                                   │
      │                                                                   │
      ▼                                                                   ▼
用户点击"连接                                                     后端使用Access Token
Google Ads"                                                       调用Google Ads API
                                                                  获取账号/广告数据
```

#### 需要的Scopes

**只读权限（推荐用于MVP）**:
```
https://www.googleapis.com/auth/adwords.readonly
```

**完整权限（创建/修改广告时需要）**:
```
https://www.googleapis.com/auth/adwords
```

#### 业务需求分析

根据AdsAI的功能设计（参考 `05-API接口设计.md`），需要以下能力：

| 功能 | API端点 | 需要的权限 |
|-----|---------|-----------|
| **导入Ads账号** | `GET /api/v1/adscenter/accounts` | ✅ 只读 |
| **同步账号数据** | `POST /api/v1/adscenter/accounts/{id}/sync` | ✅ 只读 |
| **获取广告数据** | `GET /api/v1/dashboard/overview` | ✅ 只读 |
| **预算转移** | `POST /api/v1/adscenter/transfer-budget` | ⚠️ 读写 |
| **创建广告** | `POST /api/v1/offers/{id}/deploy` | ⚠️ 读写 |

**MVP阶段建议**: 使用只读权限 (`adwords.readonly`)，待功能稳定后再升级到完整权限

### 4. API Gateway CORS 配置

**任务**: 在API Gateway配置CORS

#### 当前API Gateway配置分析

**现有配置文件**:
```
deployments/api-gateway/gateway.yaml
deployments/gateway/gateway.v2.yaml
```

**当前配置特点**:
- ✅ 已定义Firebase JWT验证
- ✅ 已配置多个微服务路由
- ❌ **缺少CORS配置**

#### CORS配置方案

Google Cloud API Gateway的CORS配置需要在OpenAPI 2.0规范中添加：

**方案A: 在OpenAPI规范中添加全局CORS**（推荐）

修改 `deployments/api-gateway/gateway.yaml`，在文件开头添加：

```yaml
swagger: "2.0"
info:
  title: adsai-gateway
  version: 1.0.0
schemes:
  - https
produces:
  - application/json

# ============ 新增 CORS 配置 ============
x-google-management:
  cors:
    allowOrigins:
      - "https://preview.example.com"
      - "https://www.example.com"
    allowMethods:
      - GET
      - POST
      - PUT
      - PATCH
      - DELETE
      - OPTIONS
    allowHeaders:
      - Authorization
      - Content-Type
      - X-Requested-With
      - X-Firebase-ID-Token
    exposeHeaders:
      - X-RateLimit-Limit
      - X-RateLimit-Remaining
      - X-RateLimit-Reset
    maxAge: 3600
    allowCredentials: true
# ========================================

securityDefinitions:
  firebase:
    type: oauth2
    # ... 现有配置
```

**方案B: 为每个路径添加OPTIONS方法**

为每个需要CORS的路径添加OPTIONS端点：

```yaml
paths:
  /api/v1/offers:
    options:
      summary: CORS preflight for /api/v1/offers
      operationId: corsOffersOptions
      responses:
        '204':
          description: CORS preflight response
          headers:
            Access-Control-Allow-Origin:
              type: string
            Access-Control-Allow-Methods:
              type: string
            Access-Control-Allow-Headers:
              type: string
            Access-Control-Max-Age:
              type: integer
    get:
      # ... 现有配置
    post:
      # ... 现有配置
```

**推荐方案**: 方案A（全局配置）更简洁高效

### 5. Firestore数据同步方案评估

**任务**: 理解Firestore工作机制，评估数据同步方案

#### Firestore工作机制分析

**1. Firestore的核心特性**

```
┌─────────────────────────────────────────────────────────┐
│                    Firestore 特性                        │
├─────────────────────────────────────────────────────────┤
│ ✅ 实时监听 (onSnapshot)                                │
│    - 数据变更自动推送到客户端                            │
│    - WebSocket长连接                                     │
│    - 支持多Tab同步                                       │
│                                                          │
│ ✅ 离线持久化                                            │
│    - 本地缓存 (persistentLocalCache)                    │
│    - 离线写入队列                                        │
│    - 自动同步                                            │
│                                                          │
│ ✅ 强一致性                                              │
│    - 单文档级别的ACID事务                                │
│    - 跨文档批量写入                                      │
│                                                          │
│ ❌ 查询能力弱                                            │
│    - 不支持复杂JOIN                                      │
│    - 不支持全文搜索                                      │
│    - 索引限制多                                          │
└─────────────────────────────────────────────────────────┘
```

**2. 当前前端已集成Firestore**

根据代码分析：

```typescript
// apps/frontend/src/core/firebase/components/FirebaseFirestoreProvider.tsx
firestore = initializeFirestore(app, {

```

**已具备能力**:
- ✅ Firestore已初始化
- ✅ 本地持久化缓存已启用
- ✅ 多Tab同步已启用
- ✅ 可直接使用 `onSnapshot` 监听数据变更

#### 数据同步方案评估

根据AdsAI的业务特点，分析三种数据同步方案：

---

**方案1: PostgreSQL主库 + Firestore缓存（原提案）**

```
PostgreSQL (主数据)
     │
     │ (Trigger)
     ▼
  Pub/Sub
     │
     │ (Cloud Function)
     ▼
Firestore (缓存)
     │
     │ (onSnapshot)
     ▼
  前端实时更新
```

**优点**:
- ✅ PostgreSQL作为单一数据源（Single Source of Truth）
- ✅ 支持复杂查询和事务
- ✅ Firestore提供实时推送能力
- ✅ 前端离线缓存

**缺点**:
- ❌ 架构复杂，需要维护同步机制
- ❌ 数据延迟（PostgreSQL → Pub/Sub → Cloud Function → Firestore，约500ms-2s）
- ❌ 数据一致性问题（同步失败时）
- ❌ 额外成本（Pub/Sub + Cloud Functions + Firestore）

**适用场景**: 需要实时推送的数据（如任务进度、账号余额）

---

**方案2: PostgreSQL主库 + SWR轮询（更简单）**

```
PostgreSQL (主数据)
     ▲
     │ (HTTP API)
     │
  API Gateway
     ▲
     │ (SWR polling)
     │
  前端定时拉取
```

**优点**:
- ✅ 架构简单，无额外组件
- ✅ 数据强一致性（直接读主库）
- ✅ 成本低（无额外服务费用）
- ✅ SWR自带缓存和去重

**缺点**:
- ❌ 无实时推送（轮询间隔延迟）
- ❌ 无离线支持
- ❌ 轮询频繁时增加后端负载

**适用场景**: 大部分列表数据（Offer列表、Ads账号列表）

---

**方案3: 混合方案（推荐）**

根据数据特点选择不同同步策略：

| 数据类型 | 同步方式 | 理由 |
|---------|---------|------|
| **Offer列表** | SWR轮询（30s） | 更新频率低，对实时性要求不高 |
| **Ads账号列表** | SWR轮询（60s） | 数据量小，变更少 |
| **任务进度** | ✅ Firestore实时 | 需要实时推送进度更新 |
| **补点击进度** | ✅ Firestore实时 | 长时间运行，需要实时展示 |
| **Token余额** | ✅ Firestore实时 | 高频变更，需要立即反馈 |
| **大盘数据** | SWR轮询（5分钟） | 统计数据，实时性要求低 |
| **通知列表** | ✅ Firestore实时 | 需要实时推送新通知 |

**架构图**:

```
┌─────────────────────────────────────────────────────────────┐
│                        前端应用                              │
├─────────────────────────────────────────────────────────────┤
│  SWR轮询                              Firestore实时监听      │
│  • Offer列表 (30s)                    • 任务进度            │
│  • Ads账号 (60s)                      • 补点击进度          │
│  • 大盘数据 (5分钟)                   • Token余额           │
│                                        • 通知                │
└───────┬─────────────────────────────────────┬───────────────┘
        │                                      │
        ▼                                      ▼
   API Gateway                           Firestore
        │                                      ▲
        ▼                                      │
   PostgreSQL ──────(Trigger)───> Pub/Sub ────┘
                                      │
                                      ▼
                               Cloud Function
```

**实现复杂度评估**:

| 组件 | 工作量 | 优先级 |
|-----|-------|--------|
| **SWR轮询** | 1天 | P0（MVP必需） |
| **PostgreSQL触发器** | 0.5天 | P1（可选） |
| **Cloud Function同步** | 1天 | P1（可选） |
| **Firestore监听** | 0.5天 | P1（可选） |

**MVP阶段建议**:
- ✅ 先实现SWR轮询（覆盖所有数据）
- ⏸️ Firestore实时推送作为Phase 2优化项

**完整版建议**:
- ✅ 混合方案（SWR + Firestore）
- ✅ 仅对高价值数据（任务进度、Token余额）使用Firestore实时推送

#### 成本对比

**SWR轮询方案**:
- Cloud Run请求: ~$0.40/百万请求
- Cloud SQL查询: 包含在实例费用中
- **月成本**: ~$10-20（1万用户）

**Firestore实时推送方案**:
- Firestore读取: $0.06/10万次
- Firestore写入: $0.18/10万次
- Pub/Sub消息: $0.40/百万条
- Cloud Functions调用: $0.40/百万次
- **月成本**: ~$50-100（1万用户）

**结论**: MVP阶段使用SWR轮询性价比最高
